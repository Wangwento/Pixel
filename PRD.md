## 一、产品概述

### 1.1 产品定位

Pixel AI 是一个面向 C 端用户的 AI 内容创作平台，核心能力为 AI 图片生成（文生图 / 图生图），同时扩展视频生成和音频生成能力。目标用户为需要社交头像、朋友圈配图、短视频素材的普通用户。

### 1.2 核心价值主张

- 低门槛：预设风格模板，不会写提示词也能出好图
- 无水印：付费用户获得高清无水印原图
- 多模态：图片 + 视频 + 音频一站式创作
- 稳定可用：多厂商调度，单一 API 故障不影响服务

### 1.3 目标用户

| 用户类型 | 特征 | 核心需求 |
|---------|------|---------|
| 社交用户 | 18-35岁，频繁更换头像 | 快速生成好看的社交头像 |
| 内容创作者 | 自媒体、小红书博主 | 批量生成配图素材 |
| 尝鲜用户 | 对 AI 好奇，偶尔使用 | 免费体验，按需付费 |

## 二、系统架构

### 2.1 技术架构

```
用户前端(React) ──┐
                  ├──→ Nginx ──→ Spring Cloud Gateway ──→ 微服务集群
管理后台(React) ──┘         (JWT鉴权 + 路由)
```

### 2.2 微服务拆分

| 服务 | 职责 | 端口 |
|------|------|------|
| pixel-gateway | API 网关，JWT 鉴权，路由转发 | 8080 |
| pixel-auth | 用户认证（登录/注册/Token） | 8081 |
| pixel-user | 用户信息、额度管理、成长任务 | 8082 |
| pixel-image | 图片生成、风格模板、资产管理 | 8083 |
| pixel-video | 视频生成、视频资产 | 8084 |
| pixel-audio | 音频生成（音乐/歌词） | 8085 |
| pixel-admin | 管理后台 API | 8086 |

### 2.3 基础设施

| 组件 | 用途 |
|------|------|
| MySQL 8.0 | 持久化存储（4 个独立库） |
| Redis 7 | 缓存、分布式锁、Token 存储 |
| Nacos 2.4 | 配置中心、服务发现 |
| RocketMQ 5.3 | 异步消息（生成事件、成长事件） |
| 阿里云 OSS | 图片/视频/音频文件存储 |

## 三、功能模块详细设计

### 3.1 用户认证模块（pixel-auth）

#### 3.1.1 注册

| 项目 | 说明 |
|------|------|
| 注册方式 | 邮箱 + 验证码 |
| 必填字段 | 用户名、邮箱、密码、验证码 |
| 校验规则 | 用户名唯一、邮箱唯一、密码≥8位 |
| 注册后动作 | 创建用户记录 → 发放新人礼包（通过成长任务） |

#### 3.1.2 登录

| 方式 | 说明 |
|------|------|
| 账号密码 | 用户名/邮箱 + 密码 |
| 手机验证码 | 手机号 + 短信验证码（预留） |

#### 3.1.3 Token 机制

- 类型：JWT
- 有效期：7 天
- 存储：Redis 缓存，支持主动失效（登出）
- 刷新：前端启动时调用 `/auth/me` 刷新用户信息

#### 3.1.4 接口清单

```
POST /api/auth/register           注册
POST /api/auth/login              登录
POST /api/auth/login-phone        手机验证码登录
POST /api/auth/logout             登出
GET  /api/auth/me                 获取当前用户
GET  /api/auth/username/check     检查用户名可用性
GET  /api/auth/email/check        检查邮箱可用性
POST /api/auth/email/send-code    发送邮箱验证码
POST /api/auth/phone/send-code    发送手机验证码
```

### 3.2 用户与额度模块（pixel-user）

#### 3.2.1 额度体系（简化方案）

**核心原则：砍掉积分系统，只保留"额度"这一个货币。**

> 现有系统同时存在"积分"和"额度"两套体系，积分还能兑换额度，过于复杂。
> 简化为：用户只关心"我还能生成几张图"，一个数字搞定。

**额度来源：**

| 来源 | 说明 |
|------|------|
| 新人礼包 | 注册即送 5 次免费额度 |
| 每日签到 | 签到送 1 次额度（连续7天额外送 3 次） |
| 套餐购买 | 购买次数包，直接增加额度 |
| 运营补偿 | 管理员手动发放（生成失败补偿等） |

**额度消耗：**

| 操作 | 消耗 |
|------|------|
| 文生图（标清） | 1 次 |
| 文生图（高清 2K） | 2 次 |
| 文生图（超清 4K） | 3 次 |
| 图生图 | 2 次 |
| 视频生成 | 5 次 |
| 音乐生成 | 3 次 |

**额度消耗顺序：**
1. 免费额度（签到、新人礼包等赠送的）
2. 购买额度（付费购买的次数包）

**防刷限制：**

| 用户类型 | 每日上限 |
|---------|---------|
| 免费用户 | 10 次/天 |
| 付费用户（有购买记录） | 50 次/天 |

#### 3.2.2 签到系统

| 项目 | 说明 |
|------|------|
| 签到奖励 | 每天签到 +1 次额度 |
| 连续签到 | 连续 7 天额外 +3 次额度 |
| 断签处理 | 连续天数归零，重新计算 |
| 每日重置 | 每天 00:00 重置签到状态 |

#### 3.2.3 成长任务

保留成长任务作为引导用户完善信息的手段，但奖励统一为额度：

| 任务 | 奖励 | 一次性 |
|------|------|--------|
| 注册成功 | 5 次额度 | 是 |
| 完善资料（昵称+头像） | 2 次额度 | 是 |
| 绑定邮箱 | 2 次额度 | 是 |
| 绑定手机 | 2 次额度 | 是 |

#### 3.2.4 接口清单

```
POST /api/user/sign-in                签到
GET  /api/user/quota                  获取额度信息
POST /api/user/profile/complete       完善资料
POST /api/user/profile/bind-email     绑定邮箱
POST /api/user/profile/bind-phone     绑定手机
GET  /api/user/growth/tasks           获取成长任务列表
POST /api/user/growth/tasks/{id}/claim 领取任务奖励
```


## 四、定价策略

### 4.1 设计原则

**砍掉积分、会员等级、VIP 月卡/年卡，只保留"次数包"模式。**

理由：
1. 积分 + 额度双货币让用户困惑（"我有 200 积分和 3 次额度，到底能生成几张？"）
2. 月卡/年卡对早期产品风险大（用户买了年卡但产品还在迭代，退款纠纷多）
3. 次数包模式简单直接：花多少钱 = 能生成多少张，用户一看就懂

### 4.2 免费额度

| 项目 | 额度 |
|------|------|
| 新人礼包 | 注册送 5 次 |
| 每日签到 | 每天 +1 次 |
| 连续签到 7 天 | 额外 +3 次 |
| 完善资料 | +2 次 |
| 绑定邮箱 | +2 次 |
| 绑定手机 | +2 次 |

> 一个活跃免费用户每月约可获得 30-40 次免费额度（签到为主），足够轻度使用。

### 4.3 付费次数包

| 套餐名称 | 次数 | 价格 | 单价 | 有效期 | 定位 |
|---------|------|------|------|--------|------|
| 体验包 | 20 次 | ¥3.9 | ¥0.20 | 30 天 | 尝鲜用户，低门槛转化 |
| 基础包 | 100 次 | ¥9.9 | ¥0.10 | 90 天 | 轻度用户，性价比入门 |
| 畅享包 | 500 次 | ¥29.9 | ¥0.06 | 180 天 | 中度用户，主力套餐 |
| 专业包 | 2000 次 | ¥79.9 | ¥0.04 | 365 天 | 重度用户/小型工作室 |

### 4.4 定价依据

**成本测算：**

| 厂商 | 单次调用成本（约） |
|------|------------------|
| 腾讯混元 | ¥0.01-0.03 |
| Google Gemini | ¥0.02-0.05 |
| 智谱 AI | ¥0.02-0.04 |
| OpenAI DALL-E 3 | ¥0.10-0.30 |

加权平均成本约 ¥0.03/次（优先调度低成本厂商）。

**毛利分析：**

| 套餐 | 单价 | 成本 | 毛利率 |
|------|------|------|--------|
| 体验包 | ¥0.20 | ¥0.03 | 85% |
| 基础包 | ¥0.10 | ¥0.03 | 70% |
| 畅享包 | ¥0.06 | ¥0.03 | 50% |
| 专业包 | ¥0.04 | ¥0.03 | 25% |

> 体验包和基础包是利润主力，畅享包是走量套餐，专业包是锚定套餐（让畅享包显得更划算）。

### 4.5 额度消耗规则

| 操作类型 | 消耗次数 | 说明 |
|---------|---------|------|
| 文生图（标清 1K） | 1 次 | 基础操作 |
| 文生图（高清 2K） | 2 次 | 分辨率翻倍 |
| 文生图（超清 4K） | 3 次 | 最高画质 |
| 图生图 | 2 次 | 需要额外处理参考图 |
| 视频生成 | 5 次 | 计算资源消耗大 |
| 音乐生成 | 3 次 | 中等消耗 |
| 歌词生成 | 1 次 | 纯文本，消耗低 |

### 4.6 消耗优先级

```
用户发起生成请求
  ↓
1. 检查每日上限（免费用户 10 次/天，付费用户 50 次/天）
  ↓
2. 优先扣减免费额度（签到、新人礼包等）
  ↓
3. 免费额度不足时，扣减购买额度（按购买时间先进先出）
  ↓
4. 所有额度不足 → 提示购买次数包
```

### 4.7 退款策略

| 场景 | 处理方式 |
|------|---------|
| 生成失败（AI 接口报错） | 自动退还消耗的额度 |
| 未使用的次数包 | 购买后 24 小时内可全额退款 |
| 已部分使用 | 不支持退款 |

### 6.2 核心用户流程

#### 流程一：新用户首次体验

```
访问首页 → 浏览热门作品 → 点击"开始创作"
  → 跳转登录页 → 注册（邮箱验证码）
  → 自动获得 5 次免费额度
  → 进入工作台 → 选择风格模板 → 输入提示词 → 生成图片
  → 下载/保存到资产库
```

#### 流程二：免费用户日常使用

```
登录 → 签到（+1 额度）→ 进入工作台
  → 生成图片（消耗额度）
  → 额度不足 → 弹出充值引导 → 购买次数包 / 明天再来签到
```

#### 流程三：付费转化

```
用户额度不足 → 弹出充值弹窗
  → 展示 4 个次数包（体验包高亮推荐）
  → 选择套餐 → 微信/支付宝支付
  → 支付成功 → 额度到账 → 继续生成
```

### 6.3 关键交互设计

**生成页面布局：**
- 左侧：参数面板（模型选择、风格模板、提示词输入、尺寸、清晰度）
- 右侧：结果展示区（生成中动画 → 结果图片）
- 底部：操作栏（下载、保存到资产、提交到热门）

**额度展示：**
- 顶部导航栏常驻显示剩余额度
- 生成按钮旁显示"本次消耗 X 次额度"
- 额度不足时生成按钮变为"充值"按钮

## 八、现有系统问题与改进建议

### 8.1 需要简化的部分

| 现有设计 | 问题 | 建议 |
|---------|------|------|
| 积分 + 额度双货币 | 用户理解成本高，兑换逻辑复杂 | 砍掉积分，统一为"额度（次数）" |
| VIP 月卡/年卡 | 早期用户量少，包月风险大 | 改为次数包按需购买 |
| VIP 等级（0-3级） | 等级区分无实际意义 | 砍掉等级，只区分"免费/付费" |
| 签到积分翻倍 | 依赖积分系统 | 签到直接送额度 |
| 积分商品兑换 | 增加运营复杂度 | 砍掉 |
| 邀请奖励积分 | 依赖积分系统 | 邀请奖励改为送额度 |
| 广告观看积分 | 依赖积分系统，且广告接入复杂 | 暂时砍掉，后期再考虑 |
